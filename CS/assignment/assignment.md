> How does SET's Credential Harvester avoid detection by the victim during and after
the attack?

```
SET's Credential Harvester avoid detection by the victim in the following ways ->

1. Looks identical to the real site: The page copies branding, layout, and the login form, so nothing appears out of place to a quick glance.
2. Uses normal web flows: Submitting a login form and then seeing a familiar page  after a redirect feels like a routine login experience.
3. No immediate error signals: The page often doesn’t show obvious warnings or failures, so the user has no clear reason to suspect anything.
4. Time gap between cause and effect: Any consequences (e.g. account misuse) happen later, making it hard to link them to that login event.
```
> How would implementing HTTPS with a valid TLS certificate on the cloned site
make detection significantly harder for a typical user?

```
Implementing HTTPS with a valid TLS certificate on a cloned site makes detection harder because it creates a false sense of security for typical users. When users see the padlock icon and “https://” in the address bar, they often assume the website is trustworthy. However, HTTPS only ensures that the connection between the user and the website is encrypted, not that the website itself is legitimate. Attackers can obtain valid TLS certificates for domains they control, so a phishing site can appear secure even though it is malicious. As a result, users who rely solely on the padlock indicator may fail to notice subtle differences in the domain name or other warning signs, making the cloned site more convincing and difficult to identify.
```

> What network-level controls (DNS filtering, proxy inspection, SIEM alerts) could detect and block a credential harvesting attack before credentials are submitted?

```
Network-level controls like 
- DNS filtering can prevent users from reaching a malicious pages and detect misspelled domain names.
- A web proxy can inspect and control outbound web traffic in real time and perform TLS inspection to analyze HTTPs contents.
- SIEM alerts (Security Information and Event Management alerts.) can alert on access to newly seen domains, multiple users visiting the same suspicious URL, login POST requests to non-corporate domains and integrate with threat intelligence feeds
This Enables early detection and rapid response before widespread impact.

```

```
Technical controls provide more reliable and long-term  protection because :

Technical controls such as DNS filtering, secure web gateways, email filtering, and multi-factor authentication provide automated protection. They work continuously, do not rely on human judgment, and can block malicious websites or emails before the user interacts with them. This makes them highly reliable and scalable across an organization. However, they may fail to detect new or sophisticated phishing attacks.

On the other hand, user security awareness training helps individuals recognize phishing attempts and suspicious behavior. It is effective against social engineering techniques and helps build a security-conscious culture. However, it is less reliable because it depends on human behavior. Users may make mistakes, especially under pressure, and the effectiveness of training may decrease over time without regular reinforcement.
In comparison, technical controls provide more consistent and reliable long-term protection because they operate automatically and reduce the chances of human error
```
